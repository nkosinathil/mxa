Usage
Command line interface
The toolkit is controlled via the cli.py module. Two input modes are available:
•	Process a folder (recursively scans for supported files):
bash
python -m forensic_toolkit.cli -d /path/to/evidence -o /path/to/output
Add --no-recursive to scan only the top level.
•	Process specific files / glob patterns:
bash
python -m forensic_toolkit.cli -i "*.zip" -i "calls.xlsx" -i "chat.html" -o /path/to/output
Common options
Option	Description
-o, --output	Required. Parent directory where a timestamped session folder will be created.
--session-name	Custom name for the session folder (default: run_YYYYMMDD_HHMMSS). Use "." to write directly into the output folder.
--skip-manifest	Skip SHA 256 hashing of inputs (faster but less forensically rigorous).
--quiet	Reduce console output.
Example
bash
# Process a folder recursively
forensic-toolkit -d "D:\cases\case_123" -o "D:\cases\output"

# Process individual files (glob patterns work on Windows too)
python -m forensic_toolkit.cli -i "WhatsApp.zip" -i "calls_*.xlsx" -o "./reports"

Output Structure
Each run creates a timestamped session folder (e.g., run_20250307_143022) with the following subdirectories:
text
session_folder/
├── whatsapp/           # WhatsApp dashboards and CSV exports
│   ├── dashboard_whatsapp_...html
│   └── whatsapp_...csv
├── texts/              # SMS/text message dashboards and CSV
│   ├── dashboard_texts_...html
│   └── texts_...csv
├── calls/              # Call log dashboards and CSV
│   ├── dashboard_calls_...html
│   └── calls_...csv
├── media/              # Extracted media files (images, audio, etc.)
├── logs/               # Chain of custody and error logs
│   ├── chain_of_custody.log
│   ├── manifest.json
│   └── errors.log
└── (other files generated by custom parsers)
The HTML dashboards are fully interactive – you can filter, search, sort, and click on risk scores to see evidence.

Extending the Toolkit
The modular design makes it easy to add support for new data sources (e.g., Telegram, Signal, iMessage backups).
Adding a new parser
1.	Create a new file in parsers/, e.g., telegram_json.py.
2.	Subclass BaseParser and implement two methods:
o	can_parse(self, file_path) – return True if the file is handled by your parser.
o	parse(self, file_path, context) – return a list of records (dictionaries).
Your records should follow a standardised format so that existing analysis and dashboard code work seamlessly:
o	For messages: timestamp, date_str, chat, sender, message, source (plus any enriched fields like category, risk_pct, meter_numbers).
Use analyze_message_text() from analysis/__init__.py to automatically enrich messages.
o	For calls: timestamp, date_str, Type, Name, Number, DurationSec, Source.
3.	Register the parser in orchestrator.py by adding it to the PARSERS list.
Modifying analysis
•	Risk rules are defined in analysis/risk.py – you can add, remove, or adjust weights.
•	Meter patterns are in analysis/meter.py.
•	Message categories are in analysis/categorization.py.
Customising dashboards
The HTML generator (dashboard/html_generator.py) uses a template with placeholders. You can modify the CSS, add new chart types, or change the table structure.

Maintenance Guidelines
To keep the toolkit reliable and forensic ready:
1.	Dependency updates – Regularly update pandas, beautifulsoup4, etc., but test thoroughly – changes in parsing libraries can affect output.
2.	Test with real data – Keep a small set of representative files (WhatsApp exports, CSV call logs) to validate after changes.
3.	Chain of custody – The custody.py module logs every operation. If you modify it, ensure all log entries are written with the correct level and format.
4.	Add unit tests – Place tests in a tests/ folder (not included in this distribution). At minimum, test each parser with sample files.
5.	Document new parsers – Update this README or create a separate PARSERS.md to describe the expected input format for each parser.
